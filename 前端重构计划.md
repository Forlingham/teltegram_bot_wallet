### Telegram 多账号处理

#### 当前机制

- **localStorage 按 TG 用户 ID 存储 Token**：`SCASH_TOKEN_<telegramUserId>`，`ensureSession()` 优先查找当前用户的缓存 Token
- **服务端支持多 Session**：`Session` 表无 userId 唯一约束，一个 User 可有多个活跃 Session（不同设备/时间）
- **数据模型是 1 用户 → 1 钱包**：Wallet 表 `userId @unique`，每个用户只能有一个钱包
- **AuthNonce 防重放**：每次登录将 Telegram `hash` 作为 nonce 存储，同一 initData 不能二次使用
- **401 自动重试**：`request()` 遇到 401 时清除当前用户 Token 重新 `ensureSession()`

#### 存在的问题

1. **无登出按钮** — settings 页面有"解绑钱包"但没有"退出登录"，后端 `POST /api/auth/logout` 存在但前端从未调用
2. **无过期 Session 清理** — 服务端 Session 过期后无定时任务清理 `Session` 和 `AuthNonce` 表
3. **切换用户时旧数据残留** — 只覆盖 `SCASH_SESSION_TOKEN`，不清理前一个用户的钱包缓存数据

#### Vue3 重构多账号策略

1. **Pinia Auth Store 持久化隔离**
   - 使用 `pinia-plugin-persistedstate`，storage key 包含当前 `telegramId`
   - 切换用户时（检测到 TG userId 变化），自动清除内存中的 wallet/price 数据，重新加载新用户的数据
   - 登出时调用 `POST /api/auth/logout` 吊销服务端 Session，并清除本地存储

2. **用户上下文切换流程**
   ```
   TG 用户打开 Mini App
     → 读取 initData 获取 telegramId
     → 检查 localStorage 中是否有该用户的 SCASH_TOKEN_<id>
       → 有: 直接使用，设置到 x-session-token 请求头
       → 无: 调用 /api/auth/telegram/login 获取新 Token，存储到 SCASH_TOKEN_<id>
     → 如果 telegramId 与上次不同，清除所有 Pinia store 内存状态，重新请求 /api/wallet/home
   ```

3. **单钱包限制的 UI 体现**
   - 如果用户已有钱包，隐藏"创建钱包"和"导入钱包"入口
   - 观察钱包用户限制发送/签名类操作（由路由守卫处理）

### Pinia Store 规划

| Store | 职责 | 持久化 |
|-------|------|--------|
| **useAuthStore** | 会话Token、TG用户信息、登录登出、自动刷新Token | `sessionToken`, `tgUserId` |
| **useWalletStore** | 钱包地址、余额、UTXO列表、是否观察钱包、备份状态 | 无 |
| **usePriceStore** | 价格数据、7日走势图、缓存策略(30min TTL) | `priceData`, `priceCacheTime` |
| **useNetworkStore** | 网络配置(bech32, pubKeyHash等)、poolAddress、arrFeeAddress | 无(由服务端注入) |

1. **send.ejs** — PSBT交易构建涉及复杂的比特币库操作，需独立composable

3. **redpacket/claim.ejs** — 独立外部页面(无底部导航，用于分享链接)，需单独路由layout
4. **inscribe.ejs** — 内容编辑+markdown预览+签名广播，需composable封装
